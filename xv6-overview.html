内核分层架构

xv6 内核是一摞依赖层:上层依赖下层,越往下越贴近硬件。把这张图记在脑子里,看任何一个文件都知道它处在哪一层、为谁服务。

系统调用 核心子系统 设备驱动 结构 / 并发 / 启动 硬件

四条关键的跨层依赖(bug 高发区)

  • kalloc.c 是 vm.c 的地基 —— 页表每一级也都用 kalloc() 分配(见 walk)。
  • trap.c 是十字路口 —— 上接 syscall.c,左接 proc.c 的 yield,下接 plic.c 设备中断。
  • 任何一次写磁盘都必须夹在 begin_op() / end_op() 之间,走 log_write() 而非裸 bwrite()。
  • spinlock / sleeplock 不属于任何一层,横向贯穿:kalloc、bcache、proc、log、ftable 各自带锁。

文件系统:内部的六层小栈

整个 FS 是 xv6 里最精巧、最花时间的部分,自身又分六层,从磁盘往上逐层抽象。

L6sysfile.copen / read / write 系统调用,负责参数检查
L5file.c文件描述符抽象,把 inode / pipe / device 统一成 file
L4fs.cinode 分配、目录、路径解析(ialloc / bmap / dirlookup / namex)
L3log.c日志:多次写攒成一个事务,崩溃可恢复(begin_op / end_op)
L2bio.c缓冲区缓存,LRU(bread / bwrite / brelse)
L1virtio_disk.c真实(模拟)磁盘 DMA 读写,描述符环

两座大山

虚拟内存 + 文件系统合计占了近一半时间,千万别压缩。pgtbl 卡住是常态 —— 回头重读书的第 3 章,而不是硬啃代码。

性价比之王

进程 · 调度虽无 lab,却最值得精读。proc.c + swtch.S + sleep/wakeup 一旦打通,fork / exec / wait / pipe / lock 全部豁然开朗。

最省力的杠杆

做 lab 时强制跑 usertests。这套回归测试在你日后用 C++ 重写时可原样复用,验证行为一致。

三明治读法

  1. 做某个 lab 前,先读 OSTEP 对应块建立直觉(它讲"是什么、为什么")。
  2. 再读 xv6 book 对应章,看 xv6 在真实代码里如何实现(它讲"代码长什么样")。
  3. 卡在底层机制(地址翻译、链接、信号)时,回查 CSAPP 第 7 / 8 / 9 章(它是 OS 脚下的硬件底座)。
